project_01
AI Outbound Automation System
Replaces manual prospecting and email writing with a fully automated outbound pipeline from data to booked meetings.

Reps were spending 60 to 70 percent of their day on research and writing rather than selling. Leadership needed to scale outbound volume significantly without growing the team.

+
What I built

Designed and implemented a fully autonomous outbound pipeline deployed on a self-hosted server. The system ingests raw prospect data, enriches contacts via multiple API integrations, runs ICP classification and fit-scoring (rubric co-developed with sales leadership), ranks decision-makers by role priority, and generates personalized outreach using an AI model grounded in a proprietary knowledge base. CRM sequences trigger automatically with no rep involvement required. I owned the full architecture: data flow, API orchestration, prompt engineering, CRM integration, and infrastructure.

project_02
Automated Research and Email Drafting System
Cuts 45 minutes of manual prep per contact by auto-generating a research brief and three email drafts directly inside the CRM the moment a rep starts a sequence.

Every time a rep added a contact to a CRM sequence, they spent 30 to 45 minutes manually researching the account and writing tailored emails. This was the single largest time drain on the team and a direct drag on pipeline throughput.

+
What I built

Designed and implemented a real-time webhook integration between the CRM's sequence trigger and an automation pipeline. When a contact enters any sequence, the system pulls CRM fields, scrapes the prospect's website, retrieves relevant account data from a public API, and combines everything with context from the AI knowledge base. The model returns a pre-call research brief and three sequenced email drafts, each with a different angle, written in the rep's voice. I built the CRM writeback logic to populate eight custom fields per contact. End-to-end time from trigger to populated CRM: under 60 seconds.

Tools used
Apollo.io (webhooks + custom fields) n8n Claude API Firecrawl Supabase RAG Public APIs JSON schema output
Technical ownership

Designed the JSON output schema for structured LLM responses, engineered prompts for consistent voice across the sales team, and built fallback logic for incomplete enrichment data. Custom field writeback designed to fit directly into the existing sales workflow.

project_03
Prospect Scoring and List Intelligence Pipeline
Turns raw, unstructured account data into a scored, ranked, CRM-ready prospect list without any manual sorting or research.

Raw public databases required significant interpretation before accounts could be used for outbound. Reps were manually sorting through hundreds of accounts, which slowed list-building to a crawl and introduced inconsistency.

+
What I built

Designed and deployed a batch processing pipeline that ingests raw account data from CSV uploads, public APIs, and CRM exports. It applies a custom AI classification taxonomy with 8 distinct categories, scores each account against a weighted ICP rubric co-developed with sales leadership, and outputs a fully enriched file with decision-makers ranked by role and seniority. Accounts that fall below revenue or structure thresholds are flagged and excluded automatically. The contact discovery layer queries multiple enrichment APIs, deduplicates results, and ranks contacts by department relevance, eliminating manual list curation entirely.

Tools used
Claude API (classification) n8n (batch orchestration) Apollo.io Hunter.io Public data APIs openpyxl (Excel output)
Collaboration and context

Classification taxonomy and ICP scoring weights were defined in collaboration with the Head of Sales. Output format was designed for direct CRM import, reducing list prep time from hours to minutes.

project_04
AI Sales Knowledge Base (RAG System)
Gives every AI-generated output access to the company's real messaging, positioning, and sales knowledge so the content is accurate and on-brand rather than generic.

AI-generated outreach was generic and off-brand. The models had no access to the company's actual positioning, competitive angles, objection handling, or case studies, which meant the output was not usable and reps were not adopting it.

+
What I built

Architected and implemented a retrieval-augmented generation (RAG) system using Supabase with the pgvector extension as the vector store, and Voyage AI for embeddings, selected over alternatives for quality and cost efficiency. Structured the knowledge base into semantic document types including messaging frameworks, product differentiators, competitive positioning, objection responses, pricing rationale, vertical pain points, and reference case studies. Wrote a custom cosine similarity match function for retrieval and integrated the knowledge layer into every downstream AI pipeline as a context injection step. Also built a parallel interface for rep use during live calls. This system is the foundation everything else depends on. Without it, AI output lacks the specificity needed for rep trust and adoption.

Tools used
Supabase + pgvector Voyage AI (embeddings) Claude API n8n SQL (custom match fn)
Architecture decisions

Chose pgvector over dedicated vector databases like Pinecone to avoid infrastructure overhead and keep the stack composable. Built the knowledge base before scaling any outreach automation, since rep trust in the output depends on content specificity, not just model capability.

Business impact
Always-onBrand and voice consistency
1 sourceOf truth for all AI output
HighRep trust and adoption
project_05
Inbound Lead Triage and Routing System
Automatically reads every inbound reply, classifies intent, and routes the lead to the right next step in under two minutes, with no rep involvement needed for triage.

Inbound reply volume was creating a secondary bottleneck. Reps could not manually triage reply volume at scale, which led to slow response times, missed high-intent signals, and lost pipeline that the outbound system had worked to create.

+
What I built

Designed and implemented an AI-powered triage layer that monitors all inbound signals including email replies, form fills, meeting requests, and re-engagement triggers. Each signal is passed through a structured classification prompt that outputs one of five intent categories: ready to buy, interested and needs nurturing, objecting, unsubscribe or DNC, or referral. Based on the classification, the system routes automatically: high-intent leads go to the meeting scheduler, objections trigger a personalized follow-up sub-sequence, and re-engagements re-enter the pipeline with full historical context. Classification outputs structured JSON rather than free text to ensure the routing logic is deterministic and reliable. Intent categories and routing rules were co-designed with the Head of Sales.

Tools used
Claude API (classification) n8n Lemlist (sequence control) Revenue Hero (booking) Apollo.io (CRM update)
Collaboration and context

Intent categories and routing decisions were co-defined with the Head of Sales based on observed reply patterns. The routing logic was built as a configurable decision tree so categories and thresholds can be updated without modifying the core pipeline.

Business impact

// 02 · approach

How I think about this work

Every system here was built around a straightforward belief: the biggest bottleneck in a sales team is rarely skill. It is time. Specifically, the hours that skilled salespeople spend on tasks that have nothing to do with closing. The goal of every project is to take those tasks off the rep's plate entirely and redirect that capacity toward conversations that actually matter.

⚙️
Reps should only handle what requires judgment

If a task can be described in a ruleset or executed with consistent data, it should be automated. Research, email drafting, data entry, and lead triage are all examples of work that should not appear on a rep's daily list.

🧠
AI output needs real context to be usable

Generic AI output does not get adopted. I built the sales knowledge system before scaling any outreach automation, because rep trust in the output depends on specificity. The AI needs to know what the company actually says, not just how to write an email.

📐
Systems should compound, not duplicate

Each system is independently functional but designed to build on the others. The classification engine feeds the sequencing system, which feeds the brief generator, which feeds the inbound router. The architecture was built to scale without starting over.

📊
Every build decision ties to a business outcome

The automation roadmap was built in close collaboration with sales leadership and tied directly to pipeline metrics needed ahead of a fundraising milestone. Technical choices were always evaluated by their effect on rep efficiency and pipeline growth, not engineering preference.
