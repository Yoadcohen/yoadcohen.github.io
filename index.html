system_01.js
Autonomous AI SDR Pipeline

Reps were acting as manual middleware — copying data between tools, writing every email from scratch, and losing 60–70% of their day to non-selling work. Leadership needed to scale pipeline without scaling headcount.

+
What I built

Designed and implemented a fully autonomous outbound SDR system deployed on a self-hosted server. The pipeline ingests raw prospect data from public databases, enriches contacts via multiple API integrations, runs an ICP classification and fit-scoring algorithm (scoring rubric co-developed with sales leadership), discovers and ranks decision-makers by role priority, and generates hyper-personalized outreach using an LLM grounded in a proprietary RAG knowledge base. CRM sequences trigger automatically — no rep input required. I owned the full architecture: data flow, API orchestration, prompt engineering, CRM integration, and infrastructure deployment.

system_02.js
CRM Sequence Trigger → AI Brief & Email Engine

Every time a rep added a contact to a CRM sequence, they spent 30–45 minutes manually researching the account and writing 2–3 tailored emails — the single largest time sink in the team's workflow and a direct drag on pipeline throughput.

+
What I built

Designed and implemented a real-time webhook integration between the CRM's sequence trigger and an n8n automation pipeline. The moment a contact enters any sequence, the pipeline pulls CRM fields, scrapes the prospect's website, retrieves financial and operational data from a public API, and passes all context — plus retrieved chunks from the RAG knowledge base — to the Claude API. The model returns a structured pre-call research brief and 3 sequenced email drafts (each with a distinct angle). I built the CRM writeback logic to populate 8 custom fields per contact. End-to-end latency: under 60 seconds from trigger to populated CRM. Email output is grounded in the company's actual positioning — not generic AI copy.

Tools used
Apollo.io (webhooks + custom fields) n8n Claude API Firecrawl Supabase RAG Public APIs JSON schema output
Technical ownership

Designed the JSON output schema for deterministic LLM responses, engineered prompts for consistent rep voice across the team, and built fallback logic for incomplete enrichment data. Custom field writeback architecture designed in collaboration with the sales ops workflow.

system_03.js
AI Prospect Classification & Intelligence Engine

The team's target market contained structurally complex account data — raw public databases required significant interpretation before they could become actionable pipeline. Reps were manually sorting through hundreds of accounts, slowing list-building to a crawl.

+
What I built

Designed and deployed an automated batch processing pipeline that ingests raw account data (CSV uploads, public APIs, CRM exports), applies a custom AI classification taxonomy with 8 distinct categories using LLM-based reasoning, scores each account against a weighted ICP rubric co-developed with sales leadership, and outputs a fully enriched CRM-ready file with decision-makers ranked by role and seniority. Accounts below revenue or structure thresholds are flagged and excluded automatically. The contact discovery layer queries multiple enrichment APIs, deduplicates results, and ranks contacts by department relevance — eliminating manual list curation entirely.

Claude API (classification) n8n (batch orchestration) Apollo.io Hunter.io Public data APIs openpyxl (Excel output)
Cross-functional context

Classification taxonomy and ICP scoring weights defined in collaboration with the Head of Sales. Output format designed for direct CRM import, reducing rep list-prep time from hours to minutes.

system_04.js
RAG-Powered Sales Knowledge System ("AI Brain")

LLM-generated outreach was generic and off-brand — reps didn't trust it and weren't using it. The AI had no access to the company's actual positioning, differentiators, case studies, or objection handling, making all downstream automation unreliable.

+
What I built

Architected and implemented a retrieval-augmented generation (RAG) system using Supabase with the pgvector extension as the vector store, and Voyage AI for embeddings (selected over OpenAI for quality-to-cost ratio). Structured the knowledge base into semantic document types: messaging frameworks, product differentiators, competitive positioning, objection responses, pricing rationale, vertical pain points, and reference case studies. Wrote a custom cosine similarity match function for retrieval, integrated the layer into every downstream AI pipeline as a context injection step, and built a parallel interactive interface for rep use during live calls. This system is the foundation all other AI pipelines depend on — without it, generated output lacks the specificity needed for rep trust and adoption.

Tools used
Supabase + pgvector Voyage AI (embeddings) Claude API n8n SQL (custom match fn)
Architecture decision

Chose pgvector over a dedicated vector DB (Pinecone, Weaviate) to avoid infrastructure overhead and keep the stack composable. Selected Voyage AI embeddings per Anthropic's recommendation for Claude-paired pipelines. Built the knowledge base before scaling any outreach automation — rep trust in output depends on specificity.

Impact on output quality & rep adoption

system_05.js
Inbound & Warmbound Intent Classification Router

Inbound reply volume from outbound sequences was creating a secondary bottleneck — reps couldn't manually triage reply volume at scale, leading to slow response times, missed high-intent signals, and lost pipeline.

+
What I built

Designed and implemented an AI-powered intent classification layer that monitors all inbound signals — email replies, form fills, meeting requests, and re-engagement triggers. Each signal is passed to the Claude API with a structured classification prompt that outputs one of five intent categories: ready to buy, interested/needs nurture, objecting, unsubscribe/DNC, or referral. Based on classified intent, the system routes automatically: high-intent leads queue in the meeting scheduler, objections trigger a personalized follow-up sub-sequence, and re-engagements re-enter the pipeline with full historical context. Classification outputs structured JSON — not free text — to ensure deterministic routing logic. Intent categories and routing rules were co-designed with the Head of Sales.

Tools used
Claude API (classification) n8n Lemlist (sequence control) Revenue Hero (booking) Apollo.io (CRM update)
Cross-functional context

Intent categories and routing decisions co-defined with the Head of Sales based on observed reply patterns. Routing logic built as a configurable decision tree — categories and thresholds can be updated without touching the core pipeline.

// 02 · thinking & strategy

Why these systems were built

These systems were built around a deliberate GTM engineering thesis: the biggest bottleneck in B2B sales isn't rep skill — it's rep leverage. Skilled closers lose half their day to tasks that should never require a human decision. Every system was prioritized by its direct contribution to pipeline volume, conversion rate, or rep efficiency — not by technical interest alone. Build decisions were made collaboratively with sales leadership and tied explicitly to GTM milestones.

⚙️
Reps as decision-makers, not operators

Every system was designed around one rule: a rep should only touch something if it requires human judgment. Research, writing, data entry, and triage should never appear on a rep's daily to-do list.

🧠
AI grounded in real sales context before scaling

Generic AI output kills rep trust — and rep trust determines adoption. I built the RAG knowledge system before scaling any outreach automation, ensuring every generated output reflected actual positioning rather than hallucinated filler.

📐
Composable architecture, not monolithic builds

Each system is independently functional but designed to compound. The classification engine feeds the sequencing system, which feeds the brief generator, which feeds the inbound router. The stack scales without rewriting core components.

📊
Every build tied to a business outcome

The automation roadmap was built in partnership with leadership and tied directly to pipeline metrics needed ahead of a fundraising milestone. Technical decisions were always evaluated through the lens of rep efficiency and pipeline impact — not engineering preference.
